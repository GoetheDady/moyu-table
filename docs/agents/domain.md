# Domain Docs

本文件说明工程类技能在探索代码库时，应该如何读取本仓库的领域文档。

本仓库使用 single-context（单上下文）布局：仓库根目录有一个 `CONTEXT.md` 用来记录项目领域语言，根目录的 `docs/adr/` 用来记录架构决策。

术语解释：

- Domain docs：领域文档，记录项目业务概念、命名习惯、核心规则和上下文。
- single-context：单上下文，表示整个仓库共享一套领域语言和架构决策文档。
- ADR：Architecture Decision Record，架构决策记录，用来说明一个重要技术或架构选择是什么、为什么这样选。

## 探索前先读取

- 仓库根目录的 `CONTEXT.md`。
- `docs/adr/` 下与当前工作区域相关的 ADR。

如果这些文件不存在，继续工作即可。不要因为它们缺失而报错，也不要预先建议创建。生产文档的技能，例如 `/grill-with-docs`，会在术语或决策真正明确后再逐步创建它们。

## 文件结构

```text
/
├── CONTEXT.md
├── docs/adr/
│   ├── 0001-example-decision.md
│   └── 0002-example-decision.md
└── src/
```

## 使用术语表里的词

当输出中需要命名领域概念时，例如 Issue 标题、重构建议、排查假设、测试名称，应优先使用 `CONTEXT.md` 中定义过的术语。不要随意换成同义词。

如果需要的概念还没有出现在术语表中，说明可能有两种情况：当前用词不是项目已有语言，需要重新考虑；或者项目确实缺少一个新概念，可以记录给 `/grill-with-docs` 后续补充。

## 标出 ADR 冲突

如果输出内容与现有 ADR 冲突，需要明确指出，而不是静默覆盖原有决策。
